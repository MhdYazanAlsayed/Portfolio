Overview

Customer Complaints Management System

This system was designed and developed end-to-end by me for the Customer Service department at Tadawi Group. Its purpose is to centralize the intake, tracking, and resolution of complaints from visitors and patients‚Äîcovering service quality, cleanliness, staff conduct, and other issues‚Äî and to ensure that no complaint is lost or forgotten.

Purpose and Scope

The system enables customer service staff to register complaints quickly and accurately, then follow them through their full lifecycle across all company branches. It standardizes how complaints are captured, prioritized, escalated, and resolved, providing managers with visibility over performance at both branch and group level.

How It Improves Response Quality

By replacing ad-hoc or manual workflows with a structured electronic process, the platform:

  • Prevents loss or duplication of complaints through a single source of truth.
  • Tracks complaint status from creation to closure with clear timestamps and owners.
  • Sends automated alerts and reminders when resolution deadlines approach or are missed.
  • Supports priority-based handling to reduce time-to-resolution for critical issues.

Priority and Escalation

Each complaint has a priority level that drives its expected resolution time. For example, complaints marked as Critical must be addressed within a few hours. If a deadline is exceeded, the system automatically escalates the case to the appropriate managers, continuing escalation until the issue is resolved or its status is explicitly changed (e.g., ‚ÄúClosed‚Äù or ‚ÄúUnresolved‚Äù).

Visibility for Staff and Management

The platform provides role-appropriate views for both agents and managers to monitor workload and outcomes across all branches. Managers can review live dashboards and scheduled reports to evaluate response effectiveness, identify recurring issues, and measure overall customer satisfaction efforts.

Position in the Enterprise Ecosystem

This system is part of a broader, integrated operations suite at Tadawi Group. While this overview focuses on the complaints module, it aligns with other enterprise components that manage the company‚Äôs day-to-day functions. Additional modules will be referenced later.

Functional Details

Staff can perform a range of actions based on permissions granted by the company-wide Identity Server (centralized users & roles). This module is part of an integrated enterprise system.

Permissions and Access Control

  • Role-based access defined centrally via Identity Server.
  • Each action (create, assign, escalate, close, reopen, configure) is permission-gated.
  • Branch scope and department scope enforced per user.

Complaint Intake

When adding a new complaint, the staff member records:

  • Branch (auto-derived from the employee‚Äôs branch) + Department + Clinic.
  • Intake Channel:
    • Phone call
    • Social media
    • Walk-in at branch
  • Complaint Type:
    • Administrative
    • Medical
    • Service
  • Priority:
    • Critical
    • Normal
    • Medium
  • Detailed Description (rephrased clearly by the agent to capture the incident precisely).

Assignment, Ownership, and Escalation Trail

  • Each complaint is assigned to a responsible staff member for end-to-end follow-up.
  • An assignment log records everyone who worked on the complaint, including timestamps.
  • When a case is escalated, the system appends the new assignee and escalation reason to the trail.

Attachments and Comments

  • Attach supporting assets (images, PDFs, other files) to clarify the issue.
  • Threaded comments allow clarifications and notes; files can be attached within comments as well.

Timeline History

  • Every status change is recorded with exact date and time.
  • Audit-ready trail includes creator, assignee changes, escalations, and closure details.

Resolution and Closure

Upon resolution, the agent must document:

  • Actions Taken (what steps were executed?)
  • Final Solution (what solved it?)
  • Corrective Plan (preventive/long-term fixes)

If resolution fails, the failure reason must be recorded in detail for managerial review.

Reopening Closed Complaints

  • A resolved case can be reopened if the customer remains dissatisfied.
  • Status resets to the beginning while preserving the full history, comments, and attachments.

Configuration and SLA Rules

  • Admin interfaces to manage:
    • Complaint Types
    • Complaint Priorities
  • Per type/priority, define an SLA (hours) for expected resolution.
  • If SLA is exceeded, the system auto-updates status, escalates, and notifies managers (in-app and email).

Organizational Hierarchy for Smart Escalations

  • Escalation paths follow the company‚Äôs hierarchical structure of employees.
  • Smart notifications and multi-level escalations target the right managers at the right time.
  • Escalation continues until the complaint is resolved or formally closed as unresolved.

All features integrate with the central Identity Server to ensure consistent user management, security, and role-based access across the company‚Äôs systems.

    public sealed record CreateTicketCommand : IRequest< int >
    {
        public string Title { get; init; } = default!;
        public TicketMethod Method { get; init; }
        public int TypeId { get; init; }
        public int? PriorityId { get; init; }
        public int? CategoryId { get; init; }
        public int? ClinicalId { get; init; }
        public int? ResponsibleId { get; init; }
        public string Description { get; init; } = default!;
    
        // Person selector (either PersonId OR full person info)
        public int? PersonId { get; init; }
        public string? PersonFullName { get; init; }
        public string? PersonIdNumber { get; init; }
        public string? PersonPhoneNumber { get; init; }
        public string? PersonEmail { get; init; }
    }

    public sealed class CreateTicketCommandValidator : AbstractValidator< CreateTicketCommand >
    {
        public CreateTicketCommandValidator()
        {
            RuleFor(x => x.Title).NotEmpty().MinimumLength(3);
            RuleFor(x => x.Description).NotEmpty().MinimumLength(10);
            RuleFor(x => x.TypeId).GreaterThan(0);
            RuleFor(x => x.Method).IsInEnum();
    
            RuleFor(x => x)
                .Must(HasValidPersonSelector)
                .WithMessage("Provide either PersonId or full person details (FullName, IdNumber, PhoneNumber).");
    
            When(x => !string.IsNullOrWhiteSpace(x.PersonEmail), () =>
            {
                RuleFor(x => x.PersonEmail!).EmailAddress();
            });
        }
    
        private static bool HasValidPersonSelector(CreateTicketCommand cmd)
        {
            var hasId = cmd.PersonId.HasValue;
            var hasFull =
                !string.IsNullOrWhiteSpace(cmd.PersonFullName) &&
                !string.IsNullOrWhiteSpace(cmd.PersonIdNumber) &&
                !string.IsNullOrWhiteSpace(cmd.PersonPhoneNumber);
    
            return hasId || hasFull;
        }
    }

    public sealed class CreateTicketCommandHandler : IRequestHandler< CreateTicketCommand, int >
    {
        private readonly IValidator< CreateTicketCommand > _validator;
        private readonly IUserService _userService;
        private readonly AppDbContext _db; //

        public CreateTicketCommandHandler(
            AppDbContext db,
            IValidator< CreateTicketCommand > validator,
            IUserService userService)
        {
            _db = db;
            _validator = validator;
            _userService = userService;
        }

        public async Task< int > Handle(CreateTicketCommand request, CancellationToken ct)
        {
            await _validator.ValidateAndThrowAsync(request, ct);

            await EnsureReferencesExistAsync(request, ct);

            // Get creator from the current authenticated user (IUserService)
            var currentUser = await _userService.GetCurrentUserAsync();
            if (currentUser is null)
                throw new NullReferenceException("User was null !");

            var personId = await ResolvePersonIdAsync(request, ct);
            var nextNumber = await GetNextTicketNumberAsync(request.BranchId, ct);

            var dto = new CreateTicketDto
            {
                Title = request.Title,
                Method = request.Method,
                SectionId = request.SectionId,
                TypeId = request.TypeId,
                Description = request.Description,
                ResponsibleId = request.ResponsibleId,
                PriorityId = request.PriorityId,
                CategoryId = request.CategoryId,
                ClinicalId = request.ClinicalId,
                BranchId = request.BranchId,
                Number = nextNumber,
                CreatorId = currentUser.Id,
                PersonId = personId
            };

            var ticket = Ticket.Create(dto);

            // Add domain event to notify the assigned employee by email after SaveChanges completes.
            // This event will be picked by a handler (e.g., outbox/dispatcher) to send the email.
            if (ticket.ResponsibleId is int assignedId)
            {
                ticket.AddDomainEvent(new SendTicketCreationNotifyToEmployee(assignedId));
            }

            _db.Tickets.Add(ticket);

            try 
            {
                await _db.SaveChangesAsync(ct);
            }
            catch(DbUpdateException ex)
            {
                throw new ValidationException("Invalid reference or duplicate data detected. Please verify IDs and uniqueness.", ex);
            }
           

            return ticket.Id;
        }

        // This to create person if he doesn't exist.
        private async Task< int > ResolvePersonIdAsync(CreateTicketCommand req, CancellationToken ct)
        {
            if (req.PersonId is int pid)
                return pid;

            var newPerson = new Person
            {
                FullName = req.PersonFullName!.Trim(),
                IdNumber = req.PersonIdNumber!.Trim(),
                PhoneNumber = req.PersonPhoneNumber!.Trim(),
                Email = string.IsNullOrWhiteSpace(req.PersonEmail) ? null : req.PersonEmail!.Trim()
            };

            _db.Persons.Add(newPerson);
            await _db.SaveChangesAsync(ct);
            return newPerson.Id;
        }

        private Task FindExistingPersonAsync(CreateTicketCommand req, CancellationToken ct)
        {
            var idNumber = req.PersonIdNumber!.Trim();
            var email = (req.PersonEmail ?? string.Empty).Trim();
            var phone = req.PersonPhoneNumber!.Trim();

            return _db.Persons
                .Where(p =>
                    p.IdNumber == idNumber ||
                    (!string.IsNullOrEmpty(email) && p.Email == email) ||
                    p.PhoneNumber == phone)
                .OrderByDescending(p => p.IdNumber == idNumber)
                .ThenByDescending(p => !string.IsNullOrEmpty(email) && p.Email == email)
                .FirstOrDefaultAsync(ct);
        }

        private async Task GetNextTicketNumberAsync(int branchId, CancellationToken ct)
        {
            var max = await _db.Tickets
                .Where(t => t.BranchId == branchId)
                .Select(t => (int?)t.Number)
                .MaxAsync(ct);

            return (max ?? 0) + 1;
        }
    }
